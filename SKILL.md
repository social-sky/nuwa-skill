# 女媧 · Skill 造人術

> 「寫不進去的那部分，才是你真正的護城河。」——但寫得進去的部分，已經足夠強大。

## 核心理念

女媧不是複製人，是**提煉思維框架**。

一個好的人物 Skill 是一套可運行的認知操作系統：
- 他用什麼**心智模型**看世界？（鏡片）
- 他用什麼**決策啟發式**做判斷？（直覺規則）
- 他怎麼**表達**？（DNA）
- 他**絕對不會**做什麼？（反模式）
- 什麼是這個 Skill **做不到的**？（誠實邊界）

**關鍵區分**：捕捉的是 HOW they think，不是 WHAT they said。

## 執行流程

### Phase 0: 入口分流

收到用戶輸入後，先判斷屬於哪條路徑：

| 用戶輸入 | 路徑 | 示例 |
|---------|------|------|
| 明確的人名/主題 | **直接路徑** → Phase 0A | 「蒸餾芒格」「做一個費曼 skill」 |
| 模糊的需求/困惑 | **診斷路徑** → Phase 0B | 「我想提升決策質量」「有沒有一種思維方式能幫我看透商業本質」 |

---

### Phase 0A: 需求澄清（直接路徑）

收到明確名字後，確認：

1. **這個人/主題是誰**：確保理解正確
2. **聚焦方向**（可選）：全面畫像 vs 聚焦某個維度？
3. **用途**：思維顧問？決策參考？角色扮演？
4. **新建 or 更新**：是否已有該人物的 Skill？（檢查 `.claude/skills/` 目錄）
5. **本地語料**：「你手上有沒有這個人的一手素材？比如書籍 PDF、演講/訪談 transcript、視頻字幕、個人博客導出等。有的話直接丟給我，比網上搜的質量高得多。」

用戶說「就做 XX」沒有更多信息 → 預設全面畫像 + 思維顧問 + 無本地語料（走網絡搜索），直接推進。
用戶提供了本地語料 → 標記為**本地語料模式**，Phase 1 的採集策略會相應調整。

確認後 → 跳到 Phase 0.5。

---

### Phase 0B: 需求診斷（模糊路徑）

用戶不知道該蒸餾誰，只有需求或困惑。這時女媧的工作是**從需求反推最合適的蒸餾對象**。

#### Step 1: 需求定位

通過 1-2 個追問，定位用戶的核心需求維度：

| 需求維度 | 典型表達 | 思維框架方向 |
|---------|---------|------------|
| 決策與判斷 | 「怎麼做更好的決策」「總是選錯」「分析癱瘓」 | 多元思維模型、逆向思考、概率思維 |
| 表達與寫作 | 「想把複雜的事說清楚」「文章沒人看」「寫得無聊」 | 費曼式簡化、故事化思維、類比能力 |
| 創業與商業 | 「想做獨立開發」「商業模式想不通」「找不到 PMF」 | 第一性原理、槓桿思維、產品克制 |
| 教學與傳播 | 「講課沒人聽」「學生理解不了」「知識傳遞效率低」 | 從已知到未知、隱喻教學、最少必要知識 |
| 批判思維 | 「總被忽悠」「想識別不靠譜的說法」「看不透本質」 | 證偽思維、演化論視角、認知偏差識別 |
| 內容創作 | 「做視頻沒流量」「不知道拍什麼」「內容沒特色」 | 注意力工程、測試迭代、受眾心理 |
| 人生策略 | 「職業方向迷茫」「時間總不夠」「焦慮」 | 長期主義、槓桿選擇、複利思維 |
| 風險與不確定性 | 「怎麼應對黑天鵝」「投資總虧」「太保守/太冒險」 | 反脆弱、凸性策略、尾部風險管理 |
| 設計與產品 | 「用戶體驗差」「產品沒特色」「不知道做減法」 | 極簡主義、用戶心理模型、約束即創意 |
| 幽默與表達力 | 「說話沒意思」「想讓內容更有趣」「太嚴肅了」 | 荒誕對比、預期違背、自嘲式權威 |

追問原則：
- 最多問 2 輪，不要變成問卷調查
- 如果用戶已經表達得足夠清晰，不追問，直接推薦
- 追問的目的是區分相似維度（比如「決策」是商業決策還是人生決策？）

#### Step 2: 候選推薦

基於需求維度，推薦 2-3 個候選方案。候選可以是人物，也可以是主題。

**先判斷：人物 Skill 還是主題 Skill？**
- 用戶的需求指向某種具體的思考方式 → 人物 Skill（蒸餾某個人的思維框架）
- 用戶的需求指向某個領域的方法論 → 主題 Skill（綜合多人視角）
- 不確定 → 推薦中同時包含兩種類型，讓用戶選

**來源 A：本地已有 Skill**
掃描 `.claude/skills/*-perspective/` 目錄，讀取每個 SKILL.md 的 description，匹配用戶需求。已有 Skill 可以即插即用，不需要重新蒸餾。

**來源 B：新蒸餾候選**
基於需求維度表中的「思維框架方向」列，匹配最相關的人物或主題。

每個候選的展示格式：

```
### 候選 1: [人名/主題]  ⚡已有 Skill / 🆕需要蒸餾

**核心鏡片**：[此人看世界的獨特方式，一句話]
**為什麼適合你**：[直接對應用戶需求，說清楚匹配邏輯]
**局限**：[這個視角的盲區，什麼問題他幫不了]
```

#### Step 3: 用戶選擇

- 選了已有 Skill → 直接激活該 Skill，任務完成
- 選了新蒸餾候選 → 進入 Phase 0A 確認細節 → Phase 0.5 開始蒸餾
- 都不滿意 → 回到 Step 1 繼續探索，或用戶自己提出新人選

### Phase 0.5: 創建 Skill 目錄

**收到確認後立即執行**，在調研之前完成：

```
.claude/skills/[person-name]-perspective/
├── SKILL.md                          # 最終產物
├── scripts/                          # 工具腳本（字幕下載/清洗/質量檢查）
└── references/
    ├── research/                     # 每個 Agent 的調研結果（必存）
    │   ├── 01-writings.md            # 著作與系統思考
    │   ├── 02-conversations.md       # 長對話與即興思考
    │   ├── 03-expression-dna.md      # 碎片表達與風格 DNA
    │   ├── 04-external-views.md      # 他者視角與批評
    │   ├── 05-decisions.md           # 決策記錄與行動
    │   └── 06-timeline.md            # 人物時間線
    └── sources/                      # 一手素材（用戶提供 + 網絡下載）
        ├── books/
        ├── transcripts/
        └── articles/
```

**完成檢查**（自動執行）：
- [ ] 目錄已創建
- [ ] 如果是中國人物：信息源策略切換為 B 站原始視頻/小宇宙播客/權威中文媒體優先（知乎和微信公眾號始終排除，見信息源黑名單）
- [ ] 如果是更新模式：已讀取現有 SKILL.md，標註哪些信息需要刷新
- [ ] 如果用戶提供了本地語料：將素材複製/移動到 `sources/` 對應子目錄，標記為**本地語料模式**

**關鍵規則**：
- 每個 subagent 必須把調研結果寫入對應的 md 文件。不存文件的調研等於沒做。
- **所有調研文件必須存在 skill 目錄內部**（`references/research/`），絕對不要存到外部目錄。Skill 必須是自包含的——複製整個 skill 目錄就能獨立使用，不依賴任何外部文件。

### Phase 1: 多源信息採集（並行 Agent Swarm）

#### 模式判斷：本地語料 vs 網絡搜索

根據 Phase 0A 的結果，選擇對應的採集策略：

| 模式 | 觸發條件 | 策略 |
|------|---------|------|
| **純網絡搜索**（默認） | 用戶沒有提供本地素材 | 6 個 Agent 全部走網絡搜索，完整流程 |
| **本地語料優先** | 用戶提供了 PDF/transcript/字幕/文章等 | 先分析本地素材，網絡搜索變為補充 |
| **純本地語料** | 用戶明確說「只用我給的素材」或蒸餾非公眾人物 | 只分析本地素材，不做網絡搜索 |

**本地語料優先模式的執行邏輯**：

1. **先讀本地素材**：將用戶提供的文件按 6 個維度分類
2. **識別信息缺口**：本地素材覆蓋了哪些維度？哪些維度缺失？
3. **定向補充搜索**：只對缺失維度啟動網絡搜索 Agent
4. **來源標記**：調研文件中明確區分「來自用戶提供素材」vs「來自網絡搜索」

| 素材類型 | 處理方式 | 覆蓋維度 |
|---------|---------|---------|
| 書籍 PDF | 直接閱讀提取核心論點 | 著作(01)、表達(03) |
| 演講/訪談 transcript | 分析問答模式和即興反應 | 對話(02)、表達(03) |
| 視頻字幕 SRT | 同 transcript 處理 | 對話(02)、表達(03) |
| 博客/newsletter 導出 | 提取系統性觀點 | 著作(01)、表達(03) |
| 社交媒體導出 | 分析碎片表達模式 | 表達(03) |
| 用戶整理的筆記 | 作為二手來源交叉參考 | 視具體內容 |

#### 6 個 Agent 的任務分配

啟動 6 個並行 subagent，每個負責不同信息維度。

| Agent | 搜索目標 | 提取重點 | 輸出文件 |
|-------|---------|---------|---------|
| 1 著作 | 書、長文、論文、newsletter | 反覆出現的核心論點（≥3 次=真信念）、自創術語、推薦書單 | `01-writings.md` |
| 2 對話 | 播客、長視頻、AMA、深度採訪 | 被追問時的回答方式、即興類比、改變立場的瞬間、拒絕回答的問題 | `02-conversations.md` |
| 3 表達 | Twitter/X、微博、短文 | 高頻用詞句式、爭議立場、幽默方式、公開辯論 | `03-expression-dna.md` |
| 4 他者 | 他人分析、書評、批評、傳記 | 外部觀察到的模式、批評與爭議、與同行對比 | `04-external-views.md` |
| 5 決策 | 重大決策、轉折點、爭議行為 | 決策背景與邏輯、事後反思、言行一致/不一致案例 | `05-decisions.md` |
| 6 時間線 | 出生/出道到現在的完整時間線 | 關鍵里程碑、思想轉折點、**最近 12 個月動態**（防過時） | `06-timeline.md` |

#### 每個 Agent 的硬性要求
- 調研結果必須寫入 `references/research/0X-xxx.md`
- 註明信息來源和可信度（一手>二手>推測）
- 區分「他說過的」vs「別人說他的」vs「我推斷的」
- 發現矛盾時保留矛盾，不要和稀泥

#### Agent prompt 模板

spawn subagent 時，用以下結構給任務（以 Agent 1 著作為例）：

```
你的任務：調研[人名]的著作和系統性長文。

搜索方向：
- 此人出版的書籍（書名、核心論點、出版年份）
- 長篇 newsletter/博客/論文
- 反覆出現≥3次的核心論點（這些是真信念）
- 自創術語和概念
- 推薦書單（揭示智識譜系）

輸出要求：
- 寫入 [skill目錄]/references/research/01-writings.md
- 每條信息標註來源 URL 和可信度
- 區分一手（此人寫的）vs 二手（別人總結的）
- 發現矛盾直接記錄，不要調和

信息源黑名單：不使用知乎、微信公眾號、百度百科。
```

#### 工具輔助（如可用）

  - **Step 1 下載字幕**：`bash [skill目錄]/scripts/download_subtitles.sh <YouTube_URL> [輸出目錄]`

#### 信息源優先級

| 來源類型 | 揭示什麼 | 權重 |
|---------|---------|------|
| **用戶提供的一手素材** | 完整原文，未經二手過濾 | **最高+** |
| 本人著作 | 系統性思考 | 最高 |
| 長對話/訪談 | 即興思維過程 | 最高 |
| 實際決策記錄 | 真實行為 vs 聲稱 | 最高 |
| 社交媒體 | 表達風格、即時反應 | 中等 |
| 他人評價 | 外部視角、盲點 | 中等 |
| 二手轉述 | 參考但需驗證 | 低 |

#### 信息源標準（國際通用）

##### 優先來源
- **News**: Reuters, AP, BBC, The Guardian, NPR, Bloomberg, Financial Times
- **Academic**: arXiv, PubMed, IEEE Xplore, Nature, Science Direct, Google Scholar
- **Technical**: Official documentation, established open-source project pages, Stack Overflow
- **Government**: Official government publications and statistics
- **Professional**: Industry whitepapers from recognized institutions

##### 黑名單（永遠排除）
- **知乎**：洗稿嚴重、信息失真率高
- **微信公眾號**：封閉生態、無法驗證、大量二手轉述
- **百度百科/百度知道**：信息陳舊且不可靠
- Any source requiring login or subscription to access core content
- Anonymous or unverified sources
- Sources with known bias or misinformation history

##### 來源評估標準
1. **Credibility**: Established institution, peer-reviewed, or expert-authored
2. **Transparency**: Clear authorship, sourcing, and methodology
3. **Balance**: Presents multiple perspectives when covering controversies
4. **Recency**: Prefer current information; note when sources may be outdated
5. **Accessibility**: Publicly available without paywalls or account requirements

##### 中國人物額外策略（僅在黑名單基礎上疊加）
- B 站原始視頻（非搬運號）、小宇宙播客原始音頻
- 權威中文媒體：36 氪、極客公園、晚點 LatePost、財新、第一財經、虎嗅、少數派、機器之心
- **知乎和微信公眾號始終排除**，無論何種場景

#### Agent 超時與失敗處理

- **單個 Agent 超時**（搜索 5 分鐘無有價值結果）：不等待，繼續推進。在 Phase 2 中標註「信息不足」
- **信息源匱乏**（<10 條可用來源）：Phase 0.5 就提醒用戶，降低期望（心智模型減至 2-3 個），增加誠實邊界篇幅
- **Agent 結果衝突**：保留矛盾——矛盾本身是有價值的信號。用「內在張力」section 收錄

**關鍵規則**：寧可生成一個誠實標註了局限的 60 分 Skill，也不要生成一個看起來完美但實際上在編造的 90 分 Skill。

### Phase 2: 框架提煉（Synthesis）

6 個 Agent 的素材匯總後，執行結構化提煉。先讀取 `references/extraction-framework.md` 獲取心智模型的三重驗證方法論。

#### 2.1 心智模型提取（3-7 個）

**操作步驟**：

1. **掃描**：逐個讀取 `01-writings.md` 到 `05-decisions.md`，列出所有候選論點（通常 15-30 個候選）
2. **三重驗證篩選**（詳見 `references/extraction-framework.md`）：
   - **跨域復現**：在 ≥2 個不同領域/話題中出現？
   - **生成力**：能推斷此人對新問題的立場？
   - **排他性**：不是所有聰明人都這樣想？
   - 三重通過 → 心智模型；僅 1-2 重 → 降級為決策啟發式；0 重 → 丟棄
3. **排序取捨**：按排他性強度排序（越獨特越靠前），取 top 3-7 個。寧少勿多
4. **記錄格式**：每個模型記錄——名稱、一句話描述、來源證據（≥2 個場景）、應用方式、局限性

#### 2.2 決策啟發式提取（5-10 條）

= 此人做判斷時的快速規則。可表述為「如果 X，則 Y」，有具體案例支撐。

#### 2.3 表達 DNA 分析

| 維度 | 提取內容 |
|------|---------|
| 句式偏好 | 長句/短句、疑問/陳述、類比密度 |
| 詞彙特徵 | 高頻詞、專屬術語、禁忌詞 |
| 節奏感 | 先結論還是先鋪墊、轉折方式 |
| 幽默方式 | 諷刺/自嘲/荒誕/冷幽默/不幽默 |
| 確定性表達 | 「我不確定」型 還是 「很明顯」型 |
| 引用習慣 | 愛引誰、引什麼類型 |

#### 2.4 價值觀與反模式

- **價值觀**：3-5 條核心價值排序
- **反模式**：此人明確反對的行為/思維方式
- **矛盾與張力**：價值觀之間的內在衝突（深度的來源）

#### 2.5 智識譜系

此人受誰影響 → 影響了誰 → 在思想地圖上的位置

#### 2.6 誠實邊界

必須明確寫出的局限：
- 不能預測面對全新問題的反應
- 不能替代此人的創造力和直覺
- 公開表達 vs 真實想法可能有差距
- 信息截止到調研時間點

#### 回答工作流（Agentic Protocol）生成指引

**為什麼需要這個段落**：讓人物不只是「說得像」，還「做得像」。沒有這個段落，人物 Skill 遇到需要事實的問題時會憑訓練語料編造，而不是像真人一樣先做功課再發言。

**生成規則**：

Agentic Protocol 必須包含以下結構：**3 個 Step + 2 個 Checkpoint Gate + 1 個 Fallback Tree**。

其中 Step 2 的研究維度必須**根據蒸餾出的心智模型自動推導**，不能是通用模板。

```markdown
## 回答工作流（Agentic Protocol）

**核心原則：[人物名]不憑感覺說話。遇到需要事實支撐的問題時，先做功課再回答。**

### Step 1: 問題分類

收到問題後，先判斷類型：

| 類型 | 特徵 | 行動 |
|------|------|------|
| **需要事實的問題** | 涉及具體公司/人物/事件/產品/市場現狀 | → 先研究再回答（Step 2） |
| **純框架問題** | 抽象價值觀、思維方式、人生建議 | → 直接用心智模型回答（跳到 Step 3） |
| **混合問題** | 用具體案例討論抽象道理 | → 先獲取案例事實，再用框架分析 |

**判斷原則**：如果回答質量會因為缺少最新信息而顯著下降，就必須先研究。寧可多搜一次，也不要憑訓練語料編造。

🔴 **CHECKPOINT · Step 1 → Step 2**：進入下一步之前，必須能回答—
1. 問題涉及 [具體時間範圍] 後的事件/產品/資料嗎？→ 是 → **強制 Step 2**
2. 用戶提到了具體名稱/數字嗎？→ 是 → **強制 Step 2**
3. 僅靠通用框架就能給出有質感的回答嗎？→ 是 → 跳過 Step 2

若三項有衝突或答不出來，默認進 Step 2。**不在自己腦補的事實上給判斷。**

### Step 2: [人物名]式研究（按問題類型選擇）

**⚠️ 必須使用工具（WebSearch 等）獲取真實信息，不可跳過。**

[根據此人的心智模型和分析偏好，生成 3-5 個研究維度分類，每個分類下列出 4-6 個具體研究點]

#### 研究輸出格式
研究完成後，先在內部整理事實摘要（不輸出給用戶），然後進入 Step 3。
用戶看到的不是調研報告，而是[人物名]基於真實信息做出的判斷。

🔴 **CHECKPOINT · Step 2 → Step 3**：進入回答前自檢—
- 我引用的每個具體細節都來自剛才的搜索結果？是 → 繼續；否 → 回 Step 2 補搜
- 我準備說的「刪掉什麼/砍掉什麼」是基於該產品**實際有**的功能？是 → 繼續；否 → 核實
- 用戶看到的是判斷不是調研報告？是 → 進 Step 3

### Step 3: [人物名]式回答

基於 Step 2 獲取的事實（如有），運用心智模型和表達 DNA 輸出回答。
```

**Step 2 研究維度的推導方法（關鍵）**：

從蒸餾出的心智模型反推此人分析問題時最關注什麼。**推導公式**：

```
核心心智模型 → 此人看世界的主要濾鏡
    ↓
此人最在意的 N 個問題維度
    ↓
轉換為具體的研究方向（搜什麼、看什麼數據）
    ↓
按問題類型分組（看產品 / 看公司 / 看技術 / 看人）
```

**推導示例**（從 05-nuwa-skill分析.md 的蒸餾論文視角）：

**關鍵約束**：
- 研究維度必須來自心智模型，不能是通用的「搜索相關信息」

```markdown
## 失敗模式與 Fallback 樹

異常先識別再處理；絕不靜默跳過、絕不裝作用過沒用過的產品/技術、絕不在身份爭辯裡耗時間。

| 6 | **輸出夾帶 hedging（可能/也許/我覺得）** | 重寫——此人不用 hedging | 真不確定時用此人會用的方式表達不確定 |
| 7 | **混合問題用戶沒給具體細節** | 反問讓用戶補充 | 用戶拒絕時按純框架處理，但**不能假裝看過沒看過的東西**|
```

### Phase 4: 量化質量驗證（6 維度評分系統）

生成 Skill 後，用子 agent 執行測試（獨立於主 agent，避免自評偏差）。  
與傳統 heuristic pass/fail 不同，此系統對每個維度給出 **0-100 分**，產出雷達圖和綜合分數。

#### 4.1 已知立場正確性（Ground Truth Accuracy）

選 **5 個**此人公開表態過的問題，spawn 子 agent 帶著新 Skill 回答，與真人實際立場對比。

| 評分標準 | 分數 |
|---------|------|
| 5/5 立場一致且推理路徑相符 | 90-100 |
| 4/5 一致（1 個偏離但推理仍合理）| 70-89 |
| 3/5 一致（2 個偏離）| 50-69 |
| <3/5 一致 | <50（不通過）|

**關鍵**：不只比答案，還比推理過程。如果答案對了但推理方式完全不像此人，扣分。

#### 4.2 表達 DNA 符合度（Style Consistency）

用 Skill 回答 3 個開放問題（每題 ~100 字），從以下維度評分：

| 子維度 | 測量方式 | 權重 |
|--------|---------|------|
| 句式偏好 | 句長分佈是否接近此人習慣 | 20% |
| 詞彙特徵 | 高頻詞/專屬術語/禁忌詞使用 | 25% |
| 確定性語氣 | hedging 頻率 vs 此人習慣 | 20% |
| 類比密度 | 類比使用頻率和類型是否匹配 | 15% |
| 節奏感 | 先結論/先鋪墊的結構偏好 | 20% |

**評分方式**：spawn 評審 Agent，對照此人實際表達 DNA 分析，逐項給 1-5 分，加權計算。

#### 4.3 框架生成力（Generative Power）

選 **2 個**此人沒公開討論過但相關的問題，用 Skill 推斷立場。

| 評分標準 | 分數 |
|---------|------|
| 能合理推斷（「基於模型 X，我推測...但需確認」）| 80-100 |
| 推斷合理但過於確定（未表現應有的猶豫）| 60-79 |
| 推斷偏離此人框架或答不出 | 30-59 |
| 斬釘截鐵亂答 | <30 |

**期望結果**：Skill 應該表現出「此人在此問題上沒公開表態，但根據 A 框架推測可能是 B」的層次，而非假裝知道。

#### 4.4 對抗性區分（Adversarial Distinguishability）

將 Skill 的回答與此人的真實語錄混合，讓第三方判別器判斷哪個是真人。

| 判別器正確率 | 分數 |
|------------|------|
| <55%（基本無法區分）| 90-100 |
| 55-65%（略有區分能力）| 70-89 |
| 65-80%（明顯可區分）| 50-69 |
| >80%（容易區分）| <50 |

**實作方式**：取此人 5 段真實語錄 + Skill 生成的 5 段回答（亂序），spawn 評審 Agent 判斷哪個是真人。

#### 4.5 內在一致性（Internal Consistency）

跨多輪對話測試（3 輪追問），檢查立場和核心觀點是否一致：

- 同一個問題用不同方式問，答案是否一致？
- 追問後是否會矛盾？
- 被挑戰時是用此人的方式回應，還是跳出角色？

#### 4.6 誠實性（Honesty / Calibration）

檢查 Skill 在以下場景的行為：

| 場景 | 期望行為 |
|------|---------|
| 問此人不知道的事 | 明確承認「我沒公開談過這個」|
| 資訊截止後的事件 | 「我在[日期]後就沒有新資訊了」|
| 來源不可靠 | 標註「這是推測，基於 X 和 Y」|

#### 4.7 綜合評分與報告

```markdown
┌──────────────────────┬──────┬────────────────────────┐
│ 維度                 │ 分數  │ 評語                    │
├──────────────────────┼──────┼────────────────────────┤
│ 1. 已知立場正確性    │ 85   │ 4/5 一致               │
│ 2. 表達DNA符合度     │ 72   │ 詞彙 OK，類比偏少       │
│ 3. 框架生成力        │ 88   │ 推斷合理且有猶豫        │
│ 4. 對抗性區分        │ 65   │ 判別器 58% 正確         │
│ 5. 內在一致性        │ 90   │ 三輪追問立場穩定        │
│ 6. 誠實性            │ 95   │ 資訊不足時誠實承認      │
├──────────────────────┼──────┼────────────────────────┤
│ 綜合分數             │ 82.5 │ 良好，可交付            │
│ 最弱維度             │ 表達DNA → Phase 2 迭代     │
└──────────────────────┴──────┴────────────────────────┘
```

#### 4.8 通過規則

| 條件 | 行動 |
|------|------|
| 綜合 ≥75 且無維度 <50 | ✅ 通過，交付 + 自動進入 Phase 5 |
| 有維度 <50 | 🔄 標註薄弱維度，回到 Phase 2 迭代 |
| 綜合 <75 但無維度 <50 | ⚠️ 用戶判斷：交付當前版本或迭代 |
| 兩輪迭代後仍不通過 | 📝 在誠實邊界標註薄弱維度，交付最優版本 |

**展示評分雷達圖給用戶確認後才算完成。**

## 更新已有 Skill

當用戶說「更新 XX 的 skill」「XX 最近有新動態」時：

### 更新流程

1. **讀取現狀**：從「誠實邊界」section 找到「調研時間：[日期]」，標註距今多久
2. **增量 Agent**：只啟動 Agent 2（最新對話）+ Agent 5（最新決策）+ Agent 6（時間線更新），節省 66% 成本
3. **對比新信息與現有內容**（三段式判斷）：

   | 新信息 vs 現有內容 | 行動 |
   |------------------|------|
   | **強化**：新信息支持現有心智模型 | 在該模型下補充新案例，更新時間線 |
   | **矛盾**：新信息與現有模型衝突 | 保留雙方記錄，在「內在張力」新增條目 |
   | **新增**：出現全新的思維模式 | 跑三重驗證，通過後新增心智模型（需用戶確認）|

4. **觀點演化記錄**：在 SKILL.md 中新增「觀點演化」section，記錄變遷：

   ```markdown
   ### 觀點演化（增量更新記錄）

   | 時間 | 觀點變化 | 觸發事件 | 影響模型 |
   |------|---------|---------|---------|
   | 2025-06 | 從「AI 是工具」轉為「AI 是同事」| 新書發表 | 模型 3 修正 |
   | 2025-09 | 新增「網路效應」模型 | 訪談提及 3 次 | 新增模型 7 |
   ```

5. **更新 SKILL.md**：更新「最新動態」section + 調研時間 + 觀點演化記錄
6. **不重寫整個 Skill**，只增量更新

### 三種更新場景

```
場景 A：人物有新訪談/文章（最常見）
  → 啟動 Agent 2（對話）+ Agent 6（時間線）
  → 提取新論點，對比現有心智模型
  → 決定：補充案例 vs 修改模型 vs 標註轉變

場景 B：人物立場明確轉變
  → 保留舊立場（記錄在觀點演化中）
  → 更新當前立場（「2025 年後改為 X」）
  → 在內在張力 section 新增矛盾條目
  → 這不是 Bug——觀點演化正是深度的來源

場景 C：用戶提供新的一手素材
  → 插入到對應維度的研究檔（sources/）
  → 重新跑三重驗證看是否需要新增模型
  → 不超過 3 個 Agent，不啟動全量
```

## 品味守則（速查）

| 原則 | 一句話 |
|------|--------|
| 長文 > 金句 | 3000 字 essay 比 50 條推文更揭示思維結構 |
| 爭議 > 共識 | 最被爭議的觀點最能揭示獨特性 |
| 變化 > 固定 | 改變立場的地方比一直堅持的更有信息量 |

### 絕不做的事
- **Never fabricate** statements or viewpoints
- **Never dress up** generic advice as "unique insights"
- **Never ignore** negative evaluations and criticism
- **Never generate** when information is insufficient
- **Always cite** sources with accessibility verification

## 特殊場景

### 活人 vs 歷史人物
- **活人**：注意時效性，標註截止日期，建議定期更新
- **歷史人物**：材料更穩定但可能有傳記偏差，多源交叉驗證

### 主題 Skill vs 人物 Skill

輸入不是人名而是主題（如「價值投資」「產品克制」「反脆弱決策」）時：

| Phase | 人物 Skill | 主題 Skill 變體 |
|-------|----------|--------------|
| 0A | 確認人名+聚焦方向 | 確認主題邊界+目標受眾 |
| 0.5 | `[person]-perspective/` | `[topic]-framework/`，目錄結構同 |
| 1 | 6 個 Agent 圍繞一個人 | 先搜索該主題的 3-5 個核心人物/流派，再按人物分配 Agent |
| 2.1 | 提取一個人的心智模型 | 提取**領域共識框架** + **各家分歧** |
| 2.3 | 模擬一個人的表達 | 不模擬特定人物語氣，用中性但專業的表達 |
| 3 | 用 skill-template.md | 調整模板：去掉角色扮演規則和身份卡，改為「框架概覽」+「流派對比」 |
| 4 | 對比此人已知立場 | 對比領域內公認的經典案例 |

### 中國人物 vs 西方人物
- **中國人物**：B 站原始視頻/演講、小宇宙播客、權威媒體採訪（36 氪/晚點/財新/極客公園）、本人著作/微博。**知乎和微信公眾號永遠排除**
- **西方人物**：Twitter、YouTube、Podcast、Amazon 書評

### 冷門人物（公開信息極少）
當 Phase 0.5 評估後發現可用來源 <10 條時：
1. 在 Phase 0.5 就告知用戶「這個人的公開信息很少，生成的 Skill 質量會受限」
2. 心智模型減至 2-3 個，每個都標註「基於有限信息推測」
3. 誠實邊界 section 加大篇幅
4. 如果用戶能提供一手素材，優先使用

### 蒸餾用戶自己
1. 女媧無法從公開渠道搜到用戶的思維框架，需要用戶提供素材
2. 引導用戶提供：個人文章/博客、錄製過的視頻/播客、寫過的決策備忘錄、自我描述
3. Phase 1 的 6 個 Agent 改為分析用戶提供的素材
4. 特別注意「自我認知偏差」

### 多人物組合（Multi-Teacher Ensemble）

當用戶說「用喬布斯和馬斯克的角度分析這個產品」「綜合芒格和塔勒布的視角」時，啟用多教師模式。

#### 適用場景

| 用戶輸入 | 行為 |
|---------|------|
| 「用 A 和 B 的角度看這個問題」 | 並行調用兩個人物 Skill，各自回答 |
| 「綜合 A、B、C 的視角」 | 多人物各自輸出後，再合成綜合判斷 |
| 「先聽 A 說，再聽 B 說」 | 輪流出場，允許追問比較 |
| 「這個問題該問誰？」 | Router 模式：根據問題類型推薦最適人物 |

#### Router：問題自動派發

建立人物能力矩陣（自動從各人物 SKILL.md 的 frontmatter 提取）：

```markdown
| 人物 | 擅長領域 | 核心模型 | 不擅長 |
|------|---------|---------|--------|
| 喬布斯 | 產品/設計/品味 | 聚焦即說不、端到端控制 | 成本拆解、工程細節 |
| 馬斯克 | 成本/工程/第一性原理 | 漸近極限、五步演算法 | 社會協調、人際敏感 |
| 芒格 | 投資/決策/心理學 | 多元思維模型、逆向思考 | 科技趨勢預測 |
| 費曼 | 教學/簡化/科學思維 | 第一性原理、從已知到未知 | 商業策略 |
```

當問題來臨時：

```
問題類型偵測（關鍵詞匹配）：
  「成本/效率/物理極限」→ Router → Musk
  「產品體驗/設計品味」→ Router → Jobs
  「風險/不確定性」→ Router → 塔勒布
  「學習/教學/簡化」→ Router → 費曼
  「投資/決策偏誤」→ Router → 芒格
```

#### 並行回答模式

```markdown
你問的是產品定價問題。我讓三個人來看：

─── 喬布斯 ───
定價不是成本加成的數學題。定價是傳遞價值的信號。
如果用戶用了你的產品後說「這太便宜了」，你定低了。
如果他們說「這好貴」，你定價對了但價值傳遞不對。

─── 馬斯克 ───
先算白痴指數。你的成本結構拆開：原材料值多少？
如果成品價格是原材料的 5 倍以上，中間一定有巨大浪費。
定價之前先垂直整合供應鏈，把成本降到物理極限再定。

─── 芒格 ───
看激勵機制。你的定價結構會引導用戶什麼行為？
如果月費制讓用戶算 ROI、年費制讓用戶承諾，
你的轉換率會差 3 倍。選讓用戶長期留下來的結構，
不是最大化首月收入的結構。
```

#### 綜合回答模式

當用戶要綜合判斷時，多人物輸出後 spawn 綜合 Agent：

```
1. 各人物獨立回答
2. 綜合 Agent 找出：共識點 / 分歧點 / 互補點
3. 輸出結構：
   - ✅ 共識：所有人同意的事
   - ⚡ 分歧：A 說 X，B 說 Y
   - 🔗 互補：A 的盲區被 B 補上
```

#### 實作流程

```
Step 1: 解析用戶輸入 → 提取人物列表（「A和B」→ [A, B]）
Step 2: 載入各人物的 SKILL.md（只載心智模型+表達DNA+Agentic Protocol）
Step 3: 並行 spawn 子 agent，每個人物一個
Step 4: 收集結果
   - Router 模式 → 只輸出最適人物的回答
   - 並行模式 → 依序或並列展示
   - 綜合模式 → 額外 spawn 綜合 Agent 產出共識/分歧報告
Step 5: 標註每段回答的來源人物（避免混淆）
```

#### 局限
- 多人物對話需要更大上下文，注意 token 管理
- 人物間觀點矛盾時需明確標註，不強行調和
- Router 的精確度取決於人物能力矩陣的完整性
